# Build on VX6

Suggested title: Build on VX6 | Use VX6 as a Base Layer for Peer-to-Peer Apps and Distributed Systems

Suggested meta description: Build on VX6 as a base network layer for peer-to-peer apps, video sharing, collaboration tools, service meshes, distributed systems, and private infrastructure.

## VX6 as a Base Layer

VX6 can sit under your app as the network layer.

That means your app can focus on:

- UI
- state
- media
- collaboration
- workflows

while VX6 handles:

- peer reachability
- naming
- localhost sharing
- relay paths
- hidden aliases
- peer discovery from any known live node

## What You Can Build

- peer-to-peer video sharing
- private meeting tools
- live collaboration suites
- team chat
- distributed IDE workspaces
- creator review rooms
- remote control panels
- service marketplaces
- distributed compute control planes
- edge control systems

## Video and Real-Time Apps

A VX6-based video app can be built around peer-first paths.

That gives you a good base for:

- small-group video
- screen sharing
- live review sessions
- creator collaboration
- internal company calls

Why this is interesting:

- traffic can move more directly between participants
- infrastructure can stay lighter
- localhost media workers can stay local

For the right topology, this can feel faster and cleaner than meeting stacks that depend on more centralized media flow.

## Why Builders Like This Model

- local-first services become network services
- direct IPv6 remains useful
- hidden aliases exist when private routing matters
- any known live node can be the network entry point
- the mesh can grow without feeling pinned to one controller

## The Builder Story

Build local first.  
Keep services on localhost.  
Use VX6 to make them reachable, named, relay-capable, and peer-aware.
