VX6 Future Prospectus

This page describes where VX6 is going. It is a living document and will be updated as phases are completed or plans change. Dates are not committed because accuracy is more important than confidence.

1 — Scope #

The long-term goal of VX6 is to make relay infrastructure unnecessary as a default mode of internet operation. The immediate goal (Phase 1) is a working direct router that replaces Tor for use cases where relay latency and exit node exposure are the primary concerns.

Subsequent phases extend VX6 from a point-to-point tool to a general-purpose network layer: one where every device can be addressed by its public key, reached directly over IPv6, and identified without consulting any registry or authority.

3 — Development phases #

Phase 1
Complete

Direct IPv6 Router

A working peer-to-peer router with mutual cryptographic authentication, encrypted tunnelling, NAT64 support for legacy applications, and a SOCKS5 proxy interface.

Deliverables: vx6 init, vx6 connect, vx6 proxy, vx6 nat64 up, vx6 status

Open issues: Protocol fingerprinting is detectable. Peer discovery requires out-of-band exchange of IPv6 addresses. Windows support requires WSL2.

Phase 2
In Development

VX6 Mesh — Spontaneous Peer Discovery

A distributed hash table (DHT) built on top of the VX6 address space. The DHT keyspace is aligned with the IPv6 address space: each node is responsible for the DHT region corresponding to its own IPv6 address. Peer lookup reduces to finding the node closest to a target address in the DHT. No bootstrap server is required after Phase 1 nodes are reached for the first time.

Phase 2 also introduces traffic obfuscation at the transport layer to resist protocol fingerprinting and deep packet inspection.

Planned deliverables: vx6 mesh join, vx6 peers, automatic reconnection, protocol camouflage mode.

Phase 3
Planned

VX6 DNS — Decentralised Name Resolution

A naming layer that allows human-readable identifiers (e.g. alice.vx6) to resolve to VX6 IPv6 addresses. Name ownership is proved by Ed25519 signature; names are registered by publishing a signed record to the VX6 DHT. There is no registrar, no ICANN delegation, and no single point of failure. A name is owned by whoever holds the corresponding private key.

The .vx6 TLD will not require ICANN registration; it operates entirely within the VX6 network and resolves only on nodes running the VX6 DNS client.

Phase 4
Planned

VX6 Mobile — Native Android and iOS

A native implementation for Android and iOS that installs as a VPN profile. The device generates its own Ed25519 keypair, derives a permanent VX6 IPv6 address, and joins the VX6 network without a desktop intermediary. The implementation does not require a jailbreak, root, or sideloading on either platform. No application-level changes are needed for existing apps to route through VX6.

Phase 5
Long Term

VX6 Hardware — Embedded Router

A reference hardware implementation targeting low-power, commodity router hardware (OpenWRT-compatible devices, small-board computers). The goal is that routers ship with VX6 support pre-installed and enabled by default, making the direct internet the default mode of operation rather than an overlay. This phase depends on Phase 2 (mesh) and Phase 3 (DNS) being stable.

4 — Non-negotiable properties #

These properties will not be compromised for velocity, commercial opportunity, or partnership requirements:

PropertyDescription
Open specification The full VX6 protocol specification is and will remain freely available under a permissive licence. Any party may implement it, fork it, or audit it without restriction.
No central authority There will be no VX6 registry, no VX6 Inc., and no mechanism by which a third party can disable or censor the network. The protocol is the institution.
Performance as a constraint Latency is treated as a security property, not an optimisation target. Features that measurably increase latency without commensurate security benefit will not be merged.
Backwards compatibility IPv4 applications must always be able to route through VX6 via NAT64. Adoption cannot require a flag day or changes to existing applications.
Self-sovereign identity The only trust anchor in the system is a keypair that the node generates and controls. No CA, no relay operator, no bootstrap server is trusted in the final design.
Censorship resistance Traffic obfuscation and protocol camouflage are treated as first-class engineering requirements, not optional features. VX6 must remain usable in high-restriction network environments.

5 — Out of scope #

The following are explicitly not goals of VX6:

  • Full anonymity. VX6 does not hide the fact that two nodes are communicating. The IPv6 source and destination addresses of VX6 packets are visible to network observers. See the security properties table for detail.
  • Content hosting. VX6 is a transport protocol. It does not define how applications should structure their data or services.
  • Consensus or blockchain. VX6 uses a DHT for peer discovery and DNS; it does not use consensus mechanisms or a distributed ledger.
  • Cryptocurrency integration. There is no token, no incentive layer, and no payment system. Relay bandwidth is irrelevant because there are no relays.

6 — Contributing #

Phase 2 (mesh peer discovery) is the current active development focus. The following areas would benefit most from external contributors:

  • DHT implementation — A Kademlia-style DHT adapted for the VX6 address space. Experience with libp2p or similar is useful.
  • Protocol security review — Formal analysis of the Noise_XX instantiation and the address derivation scheme.
  • Traffic obfuscation — Experience with obfs4, Shadowsocks, or similar protocols is directly applicable.
  • Mobile — iOS and Android VPN profile implementation. Swift and Kotlin familiarity required.
  • Embedded systems — OpenWRT module development for Phase 5 groundwork.
