VX6 Technical Overview

This page is the technical reference for the VX6 protocol. It describes the three-layer architecture (identity derivation, mutual authentication, and packet forwarding), the rationale behind each design choice, and a complete table of cryptographic primitives and protocol parameters.

1 — Background #

Understanding why VX6 was built requires understanding the structural failures of its predecessor.

1.1 — The latency problem #

Tor routes each connection through three geographically distributed relay nodes: a guard node (known to the client), a middle relay (known to neither endpoint), and an exit node (known to the destination server). Each hop adds network propagation delay and relay queuing delay. On a typical internet connection, the resulting round-trip latency is 300–800 ms. This cannot be reduced by optimising the Tor software; it is determined by the physical distance between nodes and the queuing behaviour of relay hardware.

1.2 — The exit node problem #

The exit node is, by design, the node that decrypts the final layer of the onion and forwards traffic to the destination in cleartext (for non-HTTPS connections). Any individual or organisation operating exit nodes can observe the content and metadata of traffic flowing through them. This is not a configuration or software issue; it is an unavoidable consequence of the onion routing model.

1.3 — The bandwidth problem #

Tor relay capacity is provided by volunteers. There is no mechanism to guarantee bandwidth allocation to any particular circuit. As Tor usage has grown, per-circuit bandwidth has decreased. A relay operator can throttle or terminate services at any time. VX6 connections use a dedicated channel between two nodes with no relay contention.

2 — Design #

The foundational observation behind VX6 is that IPv6's 128-bit address space is large enough to permanently assign one address per cryptographic identity, without any risk of exhaustion. An IPv6 address is not merely a routing label — in VX6, it is a commitment to a public key. This allows a connection recipient to verify the identity of a connection initiator by checking a single Diffie-Hellman operation, without consulting any registry or trusted third party.

The three layers of VX6 build on this observation:

  1. Layer 1 defines how IPv6 addresses are derived from Ed25519 public keys.
  2. Layer 2 defines how two nodes mutually authenticate using those keys (the Noise_XX handshake).
  3. Layer 3 defines how packets are forwarded through the resulting authenticated tunnel, including NAT64 support for legacy IPv4 applications.

3 — Layer 1: Identity derivation #

3.1 — Keypair generation #

Each VX6 node generates an Ed25519 keypair. Ed25519 is a deterministic signature scheme based on Curve25519. It produces 32-byte public keys and 64-byte signatures, and is resistant to timing side-channel attacks in its reference implementation. The private key (identity.ed25519) must be kept secret; the public key is distributed as part of the Node ID string.

3.2 — Address derivation #

The VX6 IPv6 address is derived by applying SHA3-256 to the raw 32-byte public key. The first 16 bytes of the resulting hash are used as the host portion of the IPv6 address. The network prefix is the VX6 ULA range fd47:06b8::/32:

/// Derive a VX6 IPv6 address from an Ed25519 public key
fn derive_ipv6(pubkey: &Ed25519PublicKey) -> Ipv6Addr {
    let hash = sha3_256(pubkey.as_bytes());     // 32 bytes
    let host = &hash[..16];                       // first 16 bytes
    build_ipv6([0xfd, 0x47, 0x06, 0xb8, 0, 0, 0, 0], host)
    // prefix: fd47:06b8::/32 (VX6 ULA range)
}

Addresses within the range fd47:06b8:: to fd47:06b8:ffff:ffff:ffff:ffff:ffff:ffff are VX6-assigned addresses. Routing infrastructure for this range is propagated by VX6 nodes and does not require global BGP advertisement.

3.3 — Ownership verification #

To verify that a peer owns the IPv6 address it claims, a verifier:

  1. Receives the peer's claimed public key (transmitted in the Noise_XX handshake).
  2. Recomputes the expected IPv6 address by applying the derivation function above.
  3. Confirms that the computed address matches the address being dialled.

This check is performed automatically by the VX6 implementation during the handshake. No certificate chain, revocation list, or authority lookup is involved.

4 — Layer 2: Mutual authentication #

4.1 — Noise_XX handshake #

VX6 uses the Noise_XX pattern from the Noise Protocol Framework (revision 34). Noise_XX was selected because it provides:

  • Mutual authentication — both the initiator and the responder prove ownership of their static Ed25519 keypairs.
  • Forward secrecy — ephemeral Diffie-Hellman keys are generated for each session; compromise of the static private key does not expose past session traffic.
  • No trusted third party — the handshake is entirely self-contained. No certificate authority, no revocation list, no PKI.
  • Identity hiding — the initiator's static public key is transmitted in the third message, after the channel is encrypted. The initiator's identity is concealed from passive observers.

4.2 — Message flow #

Notation: e = ephemeral key, s = static key  
ee/es/se = DH operations between key types

Initiator (Alice)                          Responder (Bob)
─────────────────────────────────────────────────────────────
  e                                  →
                                     ←      e, ee, s, es
  s, se                              →
─────────────────────────────────────────────────────────────
  Handshake complete.
  Both parties have derived a shared symmetric key.
  Subsequent messages are encrypted with ChaCha20-Poly1305.

After the three-message exchange:

  • Both parties have performed three Diffie-Hellman operations (ee, es, se), each contributing to the final symmetric key.
  • Both parties have authenticated each other's static public keys.
  • Both parties have verified that the peer's static key corresponds to the dialled IPv6 address (Layer 1 ownership check).

5 — Layer 3: Packet forwarding #

5.1 — NAT64 translation #

Legacy IPv4 applications produce packets with IPv4 source and destination addresses. VX6 translates these into IPv6 addresses within the fd47:06b8::/32 prefix by embedding the 32-bit IPv4 address in the lower 32 bits of the IPv6 host portion. This is analogous to the RFC 6052 Well-Known Prefix mechanism, adapted for the VX6 ULA range.

A corresponding DNS64 component rewrites A-record responses to AAAA records, so that IPv4-only hostnames resolve to VX6-addressable IPv6 addresses. This allows legacy DNS resolution to proceed without modification to the application.

5.2 — Packet flow #

Example: a legacy IPv4 application sends a TCP packet

App output:
  SRC  192.168.1.10:5432    (IPv4)
  DST  93.184.216.34:80     (IPv4)

  ↓  VX6 intercepts via NAT64 interface (vx6nat0)
  ↓  Maps IPv4 addresses to VX6 IPv6 space

After NAT64:
  SRC  fd47:6b8::c0a8:010a:5432   (VX6 IPv6)
  DST  fd47:6b8::5db8:d822:0050   (VX6 IPv6)

  ↓  Encrypted with ChaCha20-Poly1305
  ↓  Sent directly to peer — no relay involved

At peer:
  ↓  Decrypted
  ↓  NAT64 reverse-translated to IPv4
  ↓  Forwarded to application

Total relay nodes traversed: 0

6 — Protocol specifications #

ParameterValueNotes
Transport QUIC (preferred), TCP (fallback) QUIC provides multiplexed streams over UDP with built-in TLS 1.3 session encryption. TCP is selected automatically when UDP is blocked.
Signature algorithm Ed25519 (RFC 8037) 256-bit elliptic curve. 32-byte public key, 64-byte signature. Constant-time implementation required.
Handshake pattern Noise_XX Noise Protocol Framework revision 34. Uses Curve25519 for DH, SHA-256 for hashing within the pattern.
Session cipher ChaCha20-Poly1305 (RFC 8439) AEAD. 256-bit key, 96-bit nonce. Nonce is a monotonically increasing 64-bit counter, zero-padded to 96 bits.
Identity hash SHA3-256 (FIPS 202) Used to derive the IPv6 host portion from the Ed25519 public key.
Address prefix fd47:06b8::/32 Unique Local Address (ULA) range as defined in RFC 4193. Addresses are deterministically derived; no DHCP or SLAAC is used.
Default port 9600 (TCP and UDP) IANA registration pending. Sessions are multiplexed by Node ID within the protocol framing.
NAT64 prefix fd47:06b8::/96 (lower 32 bits) IPv4 addresses are embedded in the lower 32 bits of the IPv6 host portion. DNS64 rewrites A records to point at this range.
MTU 1280 bytes (minimum IPv6 MTU) Ensures compatibility across all network paths without PMTUD. Configurable up to 9000 bytes on qualifying networks.
Nonce reuse policy Hard limit at 2⁶⁴ − 1 Connection is terminated and a new session key is negotiated before nonce exhaustion.

7 — Security properties #

PropertyProvided by VX6?Mechanism
Mutual authentication Yes Noise_XX — both parties prove ownership of their static Ed25519 private key.
Confidentiality Yes ChaCha20-Poly1305 AEAD encrypts all session traffic.
Integrity Yes Poly1305 MAC detects any bit-level tampering of ciphertext.
Forward secrecy Yes Ephemeral DH keys are generated per session and discarded after the handshake.
Initiator identity hiding Yes The initiator's static public key is transmitted only after the channel is encrypted (Noise_XX message 3).
Relay node trust N/A — no relay nodes The architecture has no relay nodes; the relay trust problem does not exist.
Exit node exposure N/A — no exit nodes Traffic terminates at the peer node. There is no exit node to compromise.
Traffic analysis (metadata) Partial The source and destination IPv6 addresses of VX6 packets are visible to network observers. Timing correlation attacks remain theoretically possible. Full traffic obfuscation is planned for Phase 2.
Censorship resistance Partial (Phase 1) VX6 uses a fixed default port (9600). Traffic obfuscation and port randomisation are planned for Phase 2.
Note: VX6 does not provide anonymity in the same sense that Tor does. A passive observer can see that two specific IPv6 addresses are communicating, even if they cannot read the content. VX6 is designed to replace Tor's relay architecture and performance characteristics — not to replicate its anonymity model entirely.
