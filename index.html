VX6 — Direct IPv6 Routing Protocol

VX6 is a direct peer-to-peer routing protocol built on IPv6. It provides authenticated, encrypted connections between two nodes without any relay infrastructure. VX6 was designed as a successor to Tor, eliminating the latency, exit node risk, and bandwidth constraints that are structurally inherent to onion routing.

Overview #

Tor was built in 2002 as an overlay network on IPv4. It achieves anonymity by routing traffic through a sequence of volunteer-operated relay nodes, encrypting each layer separately. This design carries three structural limitations that no configuration change can resolve:

  • Latency. Routing through three or more geographically distributed relay nodes introduces 300–800 ms of round-trip latency. This is a consequence of network physics and relay queuing, not a software deficiency.
  • Exit node exposure. The final relay node (the exit node) decrypts the outermost layer and forwards traffic to the destination. The operator of that node can observe and alter unencrypted traffic.
  • Bandwidth contention. All Tor circuits compete for relay capacity donated by volunteer operators. Throughput is unpredictable and typically poor for large transfers.

VX6 addresses all three at the addressing layer. IPv6's 128-bit address space (3.4 × 10³⁸ addresses) is large enough that each address can be permanently bound to a cryptographic identity — specifically, to an Ed25519 public key. When an address is derived from a public key, the holder of the private key can prove ownership of that address to any other party without involving a third party. Relay infrastructure becomes unnecessary.

How It Works #

VX6 operates in three layers: identity derivation, mutual authentication, and packet forwarding.

1.1 — Identity Derivation #

Each VX6 node generates an Ed25519 keypair. The IPv6 address of that node is computed as the first 16 bytes of SHA3-256 applied to the raw public key bytes, prefixed with the VX6 ULA range fd47:06b8::/32. Because this derivation is deterministic and public, any peer can verify address ownership by checking a signature against the key embedded in the address — no authority, registry, or certificate chain required.

1.2 — Mutual Authentication #

Connections use the Noise_XX handshake from the Noise Protocol Framework. Noise_XX provides mutual authentication (both parties prove identity) with forward secrecy. It requires no certificate authority and no revocation infrastructure. After three message exchanges, both parties hold a shared symmetric key and all subsequent traffic is protected by ChaCha20-Poly1305 (AEAD).

1.3 — Packet Forwarding #

Once a tunnel is established, VX6 runs a lightweight packet forwarder. IPv6 packets addressed to a peer's VX6 address are encrypted and sent directly. Legacy IPv4 applications are supported via a NAT64 translation layer that maps IPv4 addresses into the VX6 address space, allowing existing software to use VX6 without modification.

Features #

Direct P2P connection Connections go directly from source to destination with no intermediate nodes. There are no guard nodes, middle relays, or exit nodes.
Cryptographic identity Every node's address is derived from its public key. Identity is provable by signature; it cannot be forged without the private key.
NAT64 bridge IPv4 traffic from legacy applications is transparently translated to IPv6 and routed through the VX6 tunnel without application modification.
SOCKS5 proxy Applications that support SOCKS5 can route traffic through VX6 by pointing at 127.0.0.1:1080.
No bootstrap server Peers are dialled by IPv6 address or Node ID. No central directory or bootstrap server is consulted at any point during Phase 1 operation.
Open protocol The full protocol specification is freely available. Any party may implement, fork, or audit VX6 without a commercial licence.
